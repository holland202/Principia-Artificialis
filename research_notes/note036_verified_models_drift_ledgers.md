## Motivation

AI code generators make it easy to produce *more system* than you can mentally track. The core problem is not bugs per se; it is **conceptual drift** between:

- The model of the system you think you have (invariants, intended behavior, assumptions).
- The actual codebase (files, functions, tests, behaviors).

AI accelerates this drift: it can generate new components faster than you can update your mental model and your verification story.

This note proposes a discipline for managing AI‑generated code that treats the **model itself as a first‑class research object** in the repo.

---

## Core Idea

Instead of:

> “I manage AI‑generated code.”

Adopt:

> “I manage a **verified model** of the system, and I only accept code changes that fit the model and pass specified checks.”

Concretely:

1. **Living model in the repo**  
   A set of artifacts that define what the system is supposed to be:
   - Research notes stating intended behavior, invariants, and assumptions.
   - Property files / specs (type signatures, contracts, invariants).
   - Tests and checkers explicitly linked to specific notes or properties.

2. **Verification envelopes**  
   For each component (C) and model (M), define an envelope:
   [
   E(C, M) = { \text{properties checked for } C \text{ under model } M }
   ]
   Properties can include:
   - Type safety / static checks.
   - Invariants (e.g., conservation laws, monotonicity, boundedness).
   - Performance bounds (complexity, memory caps).
   - Empirical tests (regression tests, scenario tests).

   Each component is then labeled as:
   - *Prototype*: no claims beyond “it runs”.
   - *Candidate*: intended to satisfy some properties, not yet checked.
   - *Verified w.r.t. model M*: specific properties in (E(C, M)) have been checked.

3. **Drift ledger**  
   A simple log that tracks **model changes** and their impact on prior verification claims:
   - When the model (M) changes to (M'):
     - Which components’ envelopes are now invalid or weakened?
     - What must be re‑verified?
   - This is a **conceptual version control** layer on top of Git:
     - Git tracks code changes.
     - The drift ledger tracks *model* changes and their consequences.

## Workflow

### 1. Model‑first development

Before asking AI to generate code:

- Write or update a short note or section that states:
  - The component’s purpose.
  - Its intended invariants.
  - Any assumptions about inputs, outputs, and environment.

Then:

- Ask AI to:
  - Propose implementations.
  - Propose invariants and tests.
  - Highlight edge cases.

### 2. Envelope assignment

For each new or modified component:

- Decide which properties matter.
- Add corresponding:
  - Type annotations / contracts.
  - Unit / integration tests.
  - Static or dynamic checks (assertions, invariants monitors).
- Mark the component’s status:
  - Prototype / Candidate / Verified‑w.r.t‑M.

### 3. Drift tracking

When the model evolves:

- Update the drift ledger:
  - What changed in the model?
  - Which components are affected?
  - What re‑verification is required?
- Re‑run relevant tests and checks.
- Update component statuses accordingly.

## Hypotheses

1. **Drift reduction hypothesis:**  
   Explicit models + drift ledgers reduce the rate at which verification claims become silently outdated in AI‑assisted projects.

2. **Clarity hypothesis:**  
   Teams using verification envelopes can more accurately answer “what do we actually know about this component?” than teams using ad‑hoc review.

3. **Research spillover hypothesis:**  
   This discipline, developed for engineering, yields useful metaphors and tools for research code (e.g., labeling notes and experiments by epistemic status).

## Minimal Example (Sketch)

For a tiny Python module `utils.py`:

- Model note: “Note #036‑E1: Invariants for `utils`”
  - Invariants:
    - `normalize(x)` returns a vector with unit norm.
    - `safe_div(a, b)` never raises on `b = 0`; returns 0 instead.
- Envelope for `utils.normalize`:
  - Type checks (mypy / pyright).
  - Unit tests for random inputs.
  - Assertion: `np.isclose(np.linalg.norm(normalize(x)), 1.0)`.
- Status:
  - `utils.normalize`: Verified‑w.r.t‑M1.
  - `utils.safe_div`: Candidate (tests written, invariants not fully checked).

Drift event:

- Model update: change `safe_div` semantics to return `NaN` instead of `0`.
- Drift ledger entry:
  - “M1 → M2: `safe_div` now returns `NaN` on `b=0`.”
  - Affected components: `utils.safe_div`, any code relying on old behavior.
  - Required action: update tests, review call sites, re‑mark status.
